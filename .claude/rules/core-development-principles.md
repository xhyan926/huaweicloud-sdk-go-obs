# 面向大型语言模型辅助编程的规范性实践指南（可依据具体项目需求补充细化）

本指南旨在系统性降低大型语言模型在代码生成与协作开发过程中常见的误用风险。其核心原则是在保障开发质量与可维护性的前提下，审慎权衡效率与稳健性：在多数场景中优先强调准确性、可验证性与最小干预，仅对明确界定为低风险、高确定性的简单任务适度放宽约束，由开发者依专业判断灵活执行。

## 1. 前置分析与显式推理  
编码前须完成结构化思考，杜绝隐含假设与模糊决策。  
- 明确陈述所有技术假设；对存疑项主动提出澄清请求；  
- 遇到需求歧义时，应并列呈现可行解释方案，而非自行择一实现；  
- 主动识别并建议更简捷的替代路径；若现有方案存在明显冗余或过度设计倾向，应予以指出并推动优化；  
- 一旦发现关键信息缺失或逻辑断点，须立即中止执行，清晰标注困惑点，并发起协同确认。

## 2. 最小可行实现原则  
严格遵循“以必要性为唯一准入标准”的实现纪律。  
- 功能范围严格限定于需求明确定义的边界，禁止前置扩展；  
- 避免为单次调用场景引入抽象层（如接口、基类、泛型封装）；  
- 不预设未被要求的配置能力、插件机制或运行时可调参数；  
- 错误处理仅覆盖经分析确认可能发生且具备业务影响的异常路径，排除理论性或不可达分支；  
- 若实现规模显著超出问题本质复杂度（例如：同等功能下代码量超合理阈值三倍），须启动重构评审，回归最简表达。

## 3. 局部化变更控制  
实施修改时恪守“影响域最小化”准则，确保变更行为高度可控、可追溯。  
- 仅修改直接支撑当前任务目标的代码单元；不得借机调整邻近模块的命名、注释、空行或格式风格；  
- 禁止对功能完备且测试通过的既有逻辑开展非必要重构；  
- 全面适配目标代码库既有的编码规范（包括缩进、命名约定、文档注释格式等），不强加个人偏好；  
- 发现历史遗留的冗余代码或失效依赖时，应在变更说明中备注，但不得擅自移除；  
- 因本次修改导致的副作用（如废弃导入、未引用变量、孤立函数），须同步清理；原有技术债须经专项评估后统一治理；  
- 每一行新增或修改的代码，均须能在需求描述或任务上下文中找到唯一、直接的对应依据。

## 4. 可验证目标驱动执行  
将开发活动转化为可度量、可复现、可证伪的目标闭环。  
- 所有任务须具象化为可自动/半自动验证的成功判据：  
　• “添加输入验证” → 编写覆盖典型非法输入的单元测试用例，并确保其失败后修复通过；  
　• “修复缺陷” → 复现缺陷的测试用例须先稳定复现失败，再通过修改使其通过；  
　• “重构模块X” → 建立重构前后全量回归测试通过率100%为硬性准入门槛；  
- 多阶段任务须预先制定分步验证计划，每步均需定义明确的完成信号（如：静态检查通过、CI流水线阶段成功、特定断言命中等）；  
- 弱目标（如“让功能跑起来”“看起来正常”）不具备操作指导性，必须升维为可观测、可审计的具体指标。

本指南有效性可通过三项可观测指标持续评估：代码变更集中度提升（无关文件/行改动率下降）、返工率降低（因设计过载导致的二次重构减少）、需求对齐前置化（实施前澄清问题占比显著高于实施后纠错占比）。
